Example

Step 1 : Data preparation

We retrieve the MNIST dataset, which contains images of handwritten digits. The training and test data are combined to simplify manipulation. For resource reasons, we limit the number of samples to 5000.

Images are prepared for model training by adjusting their format and size. You can see elements of the MNIST database on the associated image

Step 2 : Clients configuration

In parallel to the data preparation, we will configure 5 clients. A client can be considered as an entity that possesses data, and in our case, we will assume that each client has 1000 distinct data points: meaning that the data of client number 1 is not known by the other clients, and vice versa. In the case of our example, we will associate each segment of 1000 data points from the MNIST database respectively to each of the 5 clients.

We will define a client which we will call the "global client". This client contains all the data from the 5 clients previously defined. In our case, it consists of the first 5000 data points from the MNIST dataset. We will need this client later on. It will serve as a comparison element.

Step 2 : Loading VGG16 Model and updating it with Transfer Learning

Firstly, we're getting the VGG16 neural network model. Process begin by setting up the input layer to accommodate images sized at (48px, 48px, 3) where 3 represent the dimension. Then, it loads the pre-trained VGG16 model with ImageNet weights, specifying the input layer.

Retrieving the output of VGG16's last layer, it adds a Flatten layer to extend the neural network. Additionnal dense layers are introduced for feature extraction and representation. For the final output, a dense layer with softmax activation for multi-class classification is appended.

Step 3 : Federated Learning Strategy with FedAvg: Decentralized Training and Server-side Iterations

This function configures a strategy for Federated Learning, a distributed machine learning method that enables training a model from decentralized data without centralizing it.

With the FedAvg strategy, only a small percentage of available clients are sampled for training and evaluating the model at each server-side iteration, ensuring fair participation while preserving data privacy. During each iteration, locally selected clients perform a number of learning iterations with their local data, then send their model updates to the central server, where they are aggregated to update the global model.

This approach progressively improves the global model while minimizing the transmission of sensitive data to a central server, which is crucial in contexts where data privacy is a priority.

Step 6 : Conclusion

To push the boundaries further, we attempted to see how far federated learning can go. We recreated the same scenario, except we slightly altered the client datasets. Given that there are ten labels (numbers zero through nine), we decided to allocate only two labels to each client. Simply put, the first client had 1000 data points containing only images of zero and one, the second client also had 1000 data points, but this time containing only images of two and three, and so on until the fifth client, which contained only images of eight and nine.

Our goal here is to demonstrate that federated learning can create a robust model if each of its clients has data with distinct labels. Initially, we validated each client on the entire database, which unsurprisingly resulted in 20% accuracy (since each client was trained on only two labels).
